최성일 Git,GitHub를 활용한 버전관리
1. git과 github의 이해 및 작업 환경 세팅
  1. 깃을 사용하는 이유
    • 계속 수정이 반복되는 경우 .. 근데 수정한 것보다 그 전께 너 낫다면 ??
    • 백업을 계속 하면 파일이 많아지기 때문에 .. 관리가 어렵다 그래서 하나의 작업폴더 안에서 수정한 내역을 버전별로 관리를 하면서 어떤 부분을 어떻게 수정했는지 확인하게 편하게 만들어 놓은 시스템
    • 수정 사항을 포인트, 시작점 별로 git이 해준다!
    • git으로 버전관리한 프로젝트를 어딘가 올려야 하는데 .. 그 곳이 github(원격저장소) 이다
  2. 깃 환경 세팅
    • git 바로가기
    • 해당 버전 다운로드 한 후 next를 계속 누르면 설치가 된다.
    • 설치가 됐으면 cmd에 "git --version"을 입력 후 잘 뜨면 성공
    • 그다음 github 바로가기 github에 가서 회원가입을 해준다
    • 회원가입 후 로그인을 해준다
    • Your repositories에 들어간다. 그리고 new를 이용해 작업 프로젝트별로 저장소 공간을 만들 수 있다.
2. git init 프로젝트 폴더 초기화
  1. 깃 초기 셋팅
    • 바탕화면에 새 폴더를 만든다. (1개는 myhome 1개는 myoffice)
    • 프로젝트 파일들을 만든다.
    • 터미널 창에 "git init"을 입력한다
    • 빈 깃 저장소가 초기화 됐다라고 뜨면 된거다.
    • 한번 더 확인하려면 해당 폴더를 들어간 후 숨김폴더 보기 옵션을 체크 하면 .git이라는 폴더가 생기면 된거다.
3. git add 변경사항 스테이지에 올리기
  1. "git status" 현재 상태 확인하기(추적 하는지 안하는지)
    • "git status" 명령어는 현재 파일의 상태를 보는 것이다. 빨간색으로 파일이뜬 이유는(No commits yet = 아직 커밋 안하고 트랙킹하고있지 않음)
    • 그러면 추적할 수 있게 상황을 만들어야 한다.
    • 깃이 버전관리 할 수 있도록 어딘가 추적할 수 있는 환경을 만들어야 하는데 이것을 스테이지에 올린다고 한다.
  2. "git add ." 현재 폴더 안쪽에 있는 모든 파일을 스테이지에 올리겠다.
    • "git add ." = 해당 안에 있는 모든 파일들을 스테이지에 올려서 추가하겠다.
    • 이 상태로 "git status"를 치면 색깔이 녹색으로 바뀌고 commit할 준비가 됐다고 뜬다.
    • 파일이 변경할 떄마다 "git add ."을 해줘야 스테이지에 올라간다.
4. 작업 폴더 원격 저장소와 연결
  1. 로컬 폴더 원격 저장소 연결하는 이유
    • 협업할 때 여러 사람과 협업해야 한다면 작업한 내역이 나만 보는게 아니라 같이 개발하는 작업자들이 공유를 해야한다.
    • 공유할 원격의 공간이 github이다.
  2. 원격 저장소와 연결하는 방법
    • 1. Github 사이트에 이동한다.
    • 2. New를 눌러서 새로운 저장소를 만들 준비를 한다.
    • 3. 저장소 제목을 만든다.
    • 4. description은 optional이라 비워도 되긴 하는데 적어두면 나중에 기억하기 쉽다
    • 5. public vs private | 말 그대로 public은 공통 private는 개인이나 연관된 사람들만
    • 6. create repositories를 눌러 저장소를 만든다
    • 7. 저장소 주소를 복사를 해둔다.
    • 8. 로컬 폴더와 저장소를 등록을 해야한다.
    • 9. 깃허브 계정과 이메일 주소를 우리의 시스템에(전역에) 등록을 해야한다.
    • 10. "git config --global" = 시스템 전역에 필드 정보값을 등록을 하겠다
    • 11. "git config --global user.name 깃허브아이디" 를 입력한다
    • 12. "git config --global user.email 깃허브아이디" 를 입력한다
    • 13. "git remote add origin 깃허브저장소주소" 를 입력한다.
    • 14. 여기까지 하면 정보가 등록이 되고 저장소 주소가 연결이 된다.
    • 15. 확인하는 방법은 .git 폴더에 들어가서 config 파일을 vscode로 열어본다. 그러면 저장소 주소가 origin으로 등록이 되고 url에 주소가 들어갔으면 성공
5. git commit 중요한 시점 기록하기
  1. 작업 버전별로 기록을 남기는 방법 ' git commit -m "남길 메시지" '
    • commit을 이용하여 되돌리기 시점을 만들 수 있다.
    • git commit -m "남길 메시지" = commit을 하고 작업자들이 알아보기 편하게 그것에 대한, 수정사항에 대한 메시지를 남긴다.
  2. "git log" = commit 내역 확인하기
    • "git log"를 압력하면 commit 내역을 확인할 수 있고 커밋을 한 날짜, 메시지 등을 확인할 수 있다.

      • q를 누르면 log화면을 나갈 수 있다 quit의 약자

6. git checkout 원하는 커밋시점 되돌리기
  1. "git checkcout 커밋아이디" 타임머신 타고 과거로 이동하듯이 commit 내역 되돌리기
    • "git log"를 입력하면 커밋 내역들이 뜬다
    • commit 옆에 id를 복사한다(노란색 부분)
    • "git checkout 커밋아이디"를 입력한다
    • 그러면 해당 시점의 파일들로 복구가 된다.
    • 여기서 "git log"를 입력하면 커밋들이 해당 시점의 이전들 밖에 생기지 않는다
    • 어떻게 보면 너무 당연하지만 ..? 다시 앞으로 돌아가려면 어떻게 해야할까
  2. "git checkout master" 최신 커밋으로 이동
    • 최신 커밋으로 이동하려면 "git checkout master" 이렇게 입력하면 된다.
  3. 타임머신 타고 과거로 돌아가서 조작하면 미래가 바뀌는듯이 ..
    • 그리고 커밋을 찍어 체크아웃으로 되돌아가면 .. 변경하면 안된다. checkout은 단지 확인만 하는 것!
    • 수정을 하려면 branch를 이용해 평행세계.. 복사본을 만들어야 한다.
7. git branch 복사본 만들기
  1. 클라이언트가 자꾸 번복을 자주한다면 ..?
    • 현재 버전을 가지고, 다른 버전도 가져야 한다. 그렇다면 어떻게 해야할까
    • 그떄 필요한게 바로 branch이다.
  2. "git branch 브랜치이름"
    • "git checkout 해당시점 커밋아이디" 입력 후 해당 버전으로 돌아가기
    • 이 과거 시점에서 평생세계를 만든다 ..
    • "git branch 브랜치이름" 을 입력을 한다.
  3. "git branch -v" 브랜치 버전 확인
    • 브랜치가 잘 만들어졌는지 확인하려면 "git branch -v"를 입력한다
  4. "git checkout 브랜치이름" 해당 브랜치로 이동하기
    • "git checkout 브랜치이름" 이러면 해당 브랜치로 이동한다
    • "git branch -v" 로 브랜치 버전을 확인한다
    • "git checkout master" 를 입력하면 다시 마스터 브랜치로 이동한다!
8. git push 원격 저장소에 작업물 업로드
  1. "git push origin master" (origin은 원격저장소)
    • "git push origin master" 이렇게 입력하면 원격 저장소에 작업물이 업로드가 된다.
  2. "git push origin 브랜치이름" 하나의 저장소 안에 여러 버전을 등록할 수 있다.
    • 이러면 branch도 같이 업로드가 된다.
  3. 작업을 하다보면 branch들이 굉장히 많아진다.
    • "git push origin --all" 현재 로컬에 있는 모든 브랜치 다 올리겠다!
  4. 만약 처음 올린다면?
    • 깃허브에서 인증을 요구한다. 요구하는 인증을 해주면 된다.
    • 만약 토큰을 요구하면은
    • API 토큰 발급받는 방법은 저장소로 가서 SETTINGS를 들어간다.
    • 왼쪽 쭉 아래 내리다 보면 Developer settings 들어가고 Personal access tokens들어간다
    • Note에 이름을 적고 reop만 체크를 해준다.
    • 그리고 토큰을 생성한다.
  5. 버전별 변경한 내용 확인하기
    • 깃허브 들어가서 commit 변경 횟수를 클릭한다
    • 파일별로 이전버전과 현재버전엔 뭐가 바꼇는지 .. 비교하면서 확인할 수 있다.
9. git clone 저장소 통째로 복사하기
  1. 집에서 작업을 하다가 다음날 가서 회사에서 작업을 해야 한다면 ?
    • 매번 번거로운 작업 할 필요 없이 git 저장소에 작업물을 올려 놓으면 clone이라는 명령어로 복사해서 가져올 수 있다.
  2. "cd" = change directory
    • "cd desktop" 으로 이동
    • "cd 해당폴더" 로 이동
    • "github에가서 code버튼을 눌러 zip으로 다운로드 해도 되고, https 주소를 복사한다.
    • "git clone 저장소주소" 를 입력한다
    • 클론을 하면 마스터 브랜치만 된다. 그렇다면?
  3. "git branch -a"
    • 원격에 있는 모든 브랜치를 확인할 수 있다.
10. 저장소 복사시 서브 브랜치 활성화하기
    • 1. "git branch -a" 로 원격 저장소 브랜치 확인하기
    • 2. "git branch origin/브랜치이름" 으로 이동
    • 3. "git branch 브랜치이름" 을 입력하기
    • 4. "git branch -v" 를 입력하면 순서가 바뀌지만 .. 해당 브랜치로 이동한건 아님
    • 5. checkout으로 이동
    • 6. "git checkout 브랜치이름" 을 입력 후 "git branch -v"를 입력하면 해당 브랜치로 이동한걸 확인할 수 있다.
11. git merge로 브랜치 병합 및 conflict 해결하기
  1. pull이라는걸 이용해서 집에있는 폴더에 다시 땅겨오는 방법
    • pull을 진행하기에 앞서서 .. master push하고 브랜치 푸쉬로 올리면 Github 창에 compare & pull request 버튼이 뜬다.
    • 클릭해 들어가면 Open a pull request가 뜨고 master 브랜치가 있고 branch가 있는데 .. 저장소는 1개지만 2가지 버전이 있는 샘이다.
    • 이럴때 branch의 commit을 master와 합치고 싶을때가 있다. 이걸 mergy라는걸 이용해서 합칠 수 있는데..
    • 베이스는 마스터 .. 합쳐지는 컴페어는 해당 branch
    • create pull request 누르면 된다.
    • 근데 합치려고 보니까 conflict 즉, 충돌이 발생했다고 뜰 수 있다.
    • 각각의 branch의 같은 위치.. 같은 코드상에 변경점이 다른게 있을 때 이 중에 무엇을 선택할 것인지 .. 물어보는 것이다
    • Resolve conflicts 버튼을 클릭한다
    • 클릭해서 들어가면 2개를 합치려다 보니까 .. 예를들어
    • master 버전에서는 body부분 color:#333 , 해당branch 버전에서는 body부분 color:#666
    • 그러면 원하는 걸로 고치면 된다. 마스터껄 쓰고싶으면 마스터꺼 남기고 지운다
    • 이건 어쩔 수 없이 수작업으로 해야한다. 그리고 저장한다. Mark as resolved 클릭
    • 이러한 방식은 css와 scss를 2번 수정해야 한다.
    • 이러한 번거로움을 피하려면 푸쉬를 올릴때 먼저 머지를 하면 된다.
